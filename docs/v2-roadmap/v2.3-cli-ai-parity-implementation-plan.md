# v2.3 CLI AI Parity Implementation Plan

## Goal

Expose the v2.2 AI-assisted organization capabilities to CLI users without weakening the existing preview-first, deterministic safety model.

v2.3 is CLI parity for the already implemented GUI AI workflow. It does not add API-key providers, secure credential storage, autonomous moves, semantic destination tokens, OCR, media analysis, or cloud model configuration.

## Current Status

| Area | Status | Notes |
|------|--------|-------|
| CLI command namespace | Complete | `smartfolder ai status`, `smartfolder ai analyze`, and `smartfolder ai draft-profile` are implemented. |
| AI settings reuse | Complete | CLI reads the existing app-local GUI preferences file and falls back to disabled defaults when preferences do not exist. |
| Provider readiness | Complete | Commands reuse `OllamaClient::check_status`, respect the disabled AI gate, and remain read-only. |
| Folder analysis | Complete | `smartfolder ai analyze <folder> [--json]` reuses deterministic scan context and persisted content-inspection settings. |
| Draft profile generation | Complete | `smartfolder ai draft-profile <folder> --prompt "..." [--save-as <profile-id>] [--json]` validates drafts before save. |
| Profile explanation | Complete | `smartfolder profiles explain <profile-id> --folder <folder> [--json]` explains saved deterministic profiles. |
| Documentation and release notes | Complete | README, roadmap, changelog, and this implementation plan are updated. |

## Delivered Commands

Commands:

```powershell
smartfolder ai status [--json]
smartfolder ai analyze <folder> [--json]
smartfolder ai draft-profile <folder> --prompt "..." [--save-as <profile-id>] [--json]
smartfolder profiles explain <profile-id> --folder <folder> [--json]
```

Shared implementation details:

1. Added a top-level `ai` CLI namespace.
2. Added `ai status`, `ai analyze`, and `ai draft-profile` dispatch and argument parsing.
3. Added `profiles explain` for AI explanation of existing saved deterministic profiles.
4. Reused `smartfolder_core::ai::AiSettings`, `OllamaClient`, `AiFolderContext`, `AiFolderAnalysis`, `AiRuleProfileDraft`, `AiRuleExplanation`, and `validate_ai_profile_draft`.
5. Loaded the shared GUI preferences document through `gui_preferences_path`.
6. Returned disabled status without probing Ollama when AI is disabled or no preferences exist.
7. Required provider readiness before scan/context work for AI analyze, draft, and explain commands.
8. Reused safe scan flags: `--include-subfolders`, `--recursive`, `--max-depth`, `--current-folder-only`, `--include-hidden`, `--include-system`, `--include-project-folders`, and `--exclude`.
9. Respected persisted `content_inspection_enabled` exactly as the GUI does.
10. Added stable human-readable output for terminal users and stable JSON output for scripts.
11. Saved draft profiles only after deterministic validation succeeds.
12. Updated CLI help text to list the new AI commands and AI scan options.
13. Added unit tests for parser behavior and provider-state labels.
14. Added integration coverage for default disabled status and disabled AI workflow commands without requiring Ollama.

## Implementation Steps

### Step 1: Harden Status Command

Status: Complete

Tasks:

- Settled on app-local persisted settings only for v2.3.
- Kept `ai status` read-only; it does not persist auto-selected models.
- Retained deterministic disabled-status integration coverage. Enabled unavailable-endpoint integration coverage is deferred because the provider timeout clamp is intentionally user-facing and not suited to a fast deterministic test.

### Step 2: Add `smartfolder ai analyze <folder> [--json]`

Status: Complete

Tasks:

- Reuse existing scan options from `analyze` where safe: `--include-subfolders`, `--recursive`, `--max-depth`, `--current-folder-only`, `--include-hidden`, `--include-system`, `--include-project-folders`, and `--exclude`.
- Build `AiFolderContext` from deterministic scan records.
- Respect persisted `content_inspection_enabled` exactly as the GUI does.
- Require provider readiness before sending analysis requests.
- Print a human-readable summary with patterns, risks, recommendation, confidence, scope, and content-inspection status.
- Support `--json` using the structured `AiFolderAnalysis` response.
- Ensure errors use the existing JSON error convention when `--json` is present.

### Step 3: Add `smartfolder ai draft-profile <folder> --prompt "..." [--save-as <profile-id>]`

Status: Complete

Tasks:

- Reuse scan/context loading from AI analyze.
- Require a non-empty prompt.
- Call `OllamaClient::draft_profile`.
- Run `validate_ai_profile_draft` before presenting or saving results.
- Print validation errors and warnings without modifying saved profiles when validation fails.
- When `--save-as` is present, write a validated TOML profile into the shared profiles directory.
- Preserve the rule safety constraints from v2.2: supported deterministic fields only, safe destination tokens only, no absolute paths, no parent traversal.

### Step 4: Add `smartfolder profiles explain <profile-id> --folder <folder>`

Status: Complete

Tasks:

- Load saved profiles using the existing profile-id path rules.
- Build folder context using the same AI context helper as analysis.
- Call `OllamaClient::explain_profile`.
- Print a terminal-friendly rule explanation with likely matches and warnings.
- Support `--json` for scripts.

### Step 5: Documentation and Release Gate

Status: Complete

Tasks:

- Update `README.md` with the final v2.3 AI CLI examples after command behavior is settled.
- Update `ROADMAP.md` implementation status as each command lands.
- Update `CHANGELOG.md` under `[Unreleased]` for each user-visible command.
- Run the standard gates: `cargo fmt --check`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test --workspace`, and `cargo doc --no-deps --lib`.
- Manually verify disabled-AI behavior, unavailable-Ollama behavior, and ready-Ollama behavior where a local Ollama setup is available.

## Safety Requirements

- AI output remains advisory only.
- CLI AI commands must not move files.
- CLI AI commands must not bypass deterministic scan, preview, validation, or apply workflows.
- No file content is read unless persisted AI content inspection is enabled.
- No prompts, file contents, or raw provider payloads should be logged by default.
- Saved profiles are only written after deterministic validation succeeds.

## Settled Decisions

| Decision | Options | Recommendation | Status |
|----------|---------|----------------|--------|
| CLI AI setting overrides | Persisted GUI settings only, per-command overrides, or both | Start with persisted settings, then add explicit overrides only if needed for scripting. | Settled: persisted settings only in v2.3 |
| Auto-selected model persistence | Read-only status or write back selected preferred model | Keep CLI status read-only to avoid surprising state changes. | Settled: read-only |
| Profile draft output format | Human summary, TOML preview, JSON, or all | Human summary plus JSON first; TOML save only after validation. | Settled: human summary and JSON; TOML is written only with `--save-as` after validation |
| Explain command location | `smartfolder profiles explain` or `smartfolder ai explain-profile` | Keep roadmap shape: `profiles explain`, because the subject is an existing deterministic profile. | Settled: `profiles explain` |
