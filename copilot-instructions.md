# Copilot Instructions for smartfolder

These instructions guide AI-assisted development in this workspace.

## Project Overview

`smartfolder` is a CLI-first folder organizer for Windows/Linux/Mac. It:
1. Scans directories to collect file metadata
2. Generates organization plans using built-in or custom rules
3. Applies plans with transaction journaling for safe undo
4. Maintains audit trails and enables rollback

**Core principle**: Prioritize correctness, transparency, and data safety.

## Mandatory Documentation Standard

**Every code addition or modification must include comprehensive documentation.**

This is non-negotiable. All code changes must satisfy these requirements:

### 1. Module-Level Documentation

Every Rust file must begin with a module doc comment (`//!`) containing:
- **Purpose**: What the module does (1-2 sentences)
- **Overview**: Key concepts, types, and functions
- **Key features**: Main capabilities
- **Example workflow**: Typical usage (if applicable)

### 2. Function Documentation

Every public function must have a doc comment with:
- **Brief description**: What it does
- **Logical flow**: Step-by-step execution (for complex functions)
- **Parameters**: What each parameter means (auto-derived in most cases)
- **Returns**: What the function returns
- **Errors**: What error conditions can occur
- **Example** (for APIs): Runnable example code

### 3. Type Documentation

Every public struct, enum, or type alias must document:
- **Purpose**: What the type represents
- **Field descriptions**: Brief explanation of each field (for structs)
- **Variant descriptions**: Explanation of each variant (for enums)
- **Invariants**: Any guarantees the type maintains
- **Example**: Show typical construction or usage

### 4. Style Requirements

- **Clear and crisp**: Convey information efficiently without verbose prose
- **To the point**: No fluff or unnecessary elaboration
- **Complete**: Every public item documented; no exceptions
- **Accurate**: Match documentation to actual implementation
- **Helpful**: Explain *why* and *what*, not just *how*

## Architecture

### Three-Phase Workflow

1. **Scanning** (`scanner.rs`): File system traversal with metadata collection
2. **Planning** (`planner.rs`): Rule matching and operation generation with conflict detection
3. **Execution** (`apply.rs`): Safe file moves with transaction journaling

### Core Modules

- `lib.rs`: Crate root with public API
- `model.rs`: Data structures (records, plans, journals)
- `scanner.rs`: Directory scanning
- `planner.rs`: Plan generation
- `apply.rs`: Plan execution
- `recovery.rs`: Undo and transaction management
- `rules.rs`: Built-in and custom rules
- `paths.rs`: Path validation and safety
- `storage.rs`: Persistent storage locations
- `error.rs`: Error types
- `cli/main.rs`: Command-line interface

### Design Constraints (DO NOT VIOLATE)

- ✓ Read file metadata (size, timestamps, type)
- ✓ Preview operations before execution
- ✓ Require explicit confirmation for moves
- ✓ Store transaction journals for audit and undo
- ✗ Never read file contents
- ✗ Never move without explicit preview and confirmation
- ✗ Never overwrite existing files
- ✗ Never allow destinations outside the selected root
- ✗ Never follow symlinks by default
- ✗ Never compromise undo/journal reliability

## Common Patterns

### Logical Flow in Functions

For complex operations, use a numbered list:

```rust
/// Apply a plan to the file system, creating a transaction journal.
///
/// # Logical Flow
///
/// 1. Verify plan has selected operations
/// 2. Create and persist transaction journal
/// 3. For each operation:
///    - Check for cancellation
///    - Skip if not selected (has conflict)
///    - Verify source hasn't changed
///    - Perform the move
///    - Record result
///    - Persist journal
/// 4. Mark transaction complete or interrupted
/// 5. Return summary
```

### Type Documentation

```rust
/// Complete plan for organizing files.
///
/// Generated by the planner phase, a plan contains:
/// - List of operations to be executed (each with conflict detection)
/// - Files that couldn't be classified
/// - Warnings and summary statistics
///
/// Plans are serializable to JSON for review before execution.
/// Operations marked as "selected" will be applied; conflicts are not selected.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PlanRecord {
    pub schema_version: u16,
    pub plan_id: String,
    pub root: PathBuf,
    pub mode: PlanMode,
    pub created_at: DateTime<Utc>,
    pub operations: Vec<PlanOperation>,
    pub ambiguous_files: Vec<PathBuf>,
    pub warnings: Vec<PlanWarning>,
    pub summary: PlanSummary,
}
```
