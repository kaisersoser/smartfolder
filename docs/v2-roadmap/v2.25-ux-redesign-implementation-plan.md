# smartfolder v2.25 UX redesign implementation plan

## Status

This plan defines the v2.25 UX redesign track for the `smartfolder-gui` desktop app.

The v2.0 UX rewrite shipped the first full Windows-first desktop workflow. The v2.25 track is a refinement and trust pass: it keeps the existing scan, preview, apply, undo, activity, rules, settings, and AI-assist behavior, but turns the current interface into a calmer, more precise, and more maintainable egui application.

## Product direction

smartfolder v2.25 should feel like a precise desktop utility:

- safe before powerful
- preview-led
- undo-visible
- calm and dense without feeling cramped
- deterministic first, AI-assisted only where useful
- simple by default, deep on demand

The guiding phrase is:

```text
Preview before anything moves. Restore after anything changes.
```

## Current implementation substrate

The redesign is adapted to the current Rust desktop app, not to a CSS or web component model.

Relevant implementation facts:

- GUI stack: `eframe` and `egui`.
- Main GUI state and rendering currently live mostly in `crates/smartfolder-gui/src/main.rs`.
- Theme primitives already exist in `crates/smartfolder-gui/src/ui/theme`.
- Semantic icon tokens already exist in `crates/smartfolder-gui/src/ui/icons.rs`.
- The app already supports:
  - Explorer-preloaded folder launch
  - folder selection and recent folders
  - built-in organization modes
  - custom rule profiles
  - SQLite-backed paged preview
  - safe apply
  - transaction-backed undo
  - Activity, Rules, and Settings sections
  - optional Ollama-backed AI assistance
- The worktree may contain in-progress AI notes and prompt-refinement UI work in `main.rs` and `icons.rs`; this plan treats that as existing work and does not require reverting it.

## Settled v2.25 UX decisions

| Area | Decision |
|---|---|
| Theme default | Keep `System` as the default preference. Refine both light and dark themes, with dark graphite as the strongest product expression. |
| Preview data model | Add core and session-store support for explicit untouched reasons. Do not fake this only in GUI copy. |
| Detailed preview | Use a hybrid model: compact inline preview on the main screen plus a full paged detailed preview window. |
| Completion `Done` behavior | Return to the Organize flow with the same folder selected, clear the completion action panel, and leave Activity as the place for prior restores. |
| Rule builder surface | Keep Profile Workspace as an `egui::Window` for early v2.25 phases. Migrate to a full Rules-page split layout later. |
| Refactoring scope | Allow phased extraction from `main.rs` into screen and component modules, while preserving behavior and keeping builds green. |
| Grill-me workflow | Future open questions should be asked one dependency branch at a time, not as a large batch. |

## Non-negotiables

- Do not weaken preview, explicit confirmation, safe destination checks, no-overwrite behavior, journaling, or undo.
- Do not make AI the primary trust mechanism.
- Do not hide failures or conflicts to make the interface look calmer.
- Do not expose sessions, journals, transactions, SQLite, or internal IDs in primary workflow surfaces.
- Do not materialize huge previews in memory; preserve SQLite paging.
- Do not break Explorer-preloaded folder launch.
- Keep CLI and shared-core behavior compatible.
- Keep the project compiling and tests passing after each implementation slice.

## Terminology direction

Primary UI copy should use product language:

| Current/internal term | Primary v2.25 UX term |
|---|---|
| Apply ready moves | Organize files |
| Undo Changes | Restore previous layout |
| Transaction | Activity |
| Journal | Restore history |
| Selected operation | Ready move |
| Ambiguous file | Untouched or needs review, depending on reason |
| Conflict | Needs review |
| Session | Hidden from primary UI |
| AppData path | Technical detail |

Technical terms may still appear in advanced panels, diagnostics, logs, and exported data.

## Information architecture

Primary navigation remains:

```text
Organize
Activity
Rules
Settings
```

The shell should use:

- a persistent left rail
- icon-only collapsed mode with tooltips
- optional expanded labels
- subtle active state
- centered max-width page content
- one obvious primary action per screen

Recommended page widths:

| Screen | Width target |
|---|---:|
| Organize flow | 960 px |
| Settings | 960 px |
| Activity | 1120 px |
| Rules list | 1120 px |
| Profile workspace | Flexible split layout |

## Implementation milestones

### UX-225-0 - Baseline and guardrails

Purpose:

Freeze the current functional baseline before changing UX structure.

Scope:

- Preserve existing uncommitted GUI work unless it directly conflicts with v2.25 changes.
- Record baseline validation results.
- Add or update fixtures that cover ready moves, destination conflicts, no matching rule, unsafe destination, and already organized or no-op files.
- Identify functions in `main.rs` that should remain behavior owners during extraction.

Acceptance criteria:

- `cargo fmt --check` passes.
- `cargo test -p smartfolder-core` passes.
- `cargo test -p smartfolder-gui` passes.
- `cargo test -p smartfolder-gui --no-run` passes during early UI-only slices.
- A disposable folder can still complete Analyze Folder -> Preview -> Organize Files -> Restore previous layout.
- Explorer-preloaded folder launch still opens with the folder selected and no file changes made.

Implementation notes:

- Do not begin visual extraction until the current baseline is buildable.
- Avoid large behavior refactors in this milestone.

### UX-225-1 - egui design foundation

Purpose:

Create a reusable egui-native design foundation so later screens can be redesigned without duplicating one-off style code.

Scope:

- Update typography tokens:
  - page title: 26 px
  - section title: 16 px
  - card title: 15 to 16 px
  - body: 14 px
  - caption: 12 px
- Update spacing and sizing tokens:
  - normal workflow controls: 36 to 40 px tall
  - compact toolbar controls: 32 to 34 px tall
  - cards and panels: 12 to 16 px padding
  - page gaps: 16 to 24 px
- Refine both light and dark palettes:
  - keep `System` as default
  - make dark graphite quiet and polished
  - reduce warm cream dominance in light mode
  - use one primary blue accent
  - reduce status color intensity
- Expand `ui/theme/widgets.rs` with:
  - primary, secondary, tertiary, and danger buttons
  - compact button variants
  - icon button helpers
  - subtle frame and raised frame helpers
  - status dot and status pill helpers
  - disclosure helper patterns

Acceptance criteria:

- Screens can express button hierarchy without local ad hoc button styling.
- Status colors are available but visually restrained.
- Most new UI work uses theme helpers instead of raw `Color32::from_rgb`.
- Focus and hover states remain visible in both light and dark themes.

### UX-225-2 - Shell, headers, and layout rhythm

Purpose:

Make the app feel like a coherent desktop product before deeper screen work begins.

Scope:

- Replace large screen headings with a reusable page-header helper.
- Use smaller page titles and shorter subtitles.
- Make the collapsed navigation rail the primary visual target.
- Add tooltips for rail icons.
- Reduce heavy active states in the rail.
- Add max-width layout helpers for Organize, Activity, Rules, and Settings.
- Replace the large block-style stepper with a compact inline stepper:

```text
Folder -> Instructions -> Preview -> Organize
```

Acceptance criteria:

- Each section has one obvious primary action.
- Page titles no longer dominate the screen.
- The rail is usable in collapsed mode.
- The Organize stepper guides the flow without consuming excessive vertical space.
- No screen requires internal implementation language to understand where the user is.

### UX-225-3 - Preview reason data model

Purpose:

Make the redesigned Preview honest by giving it real data for why files stay untouched.

Scope:

- Add an explicit core model for untouched files, for example:

```rust
UntouchedRecord {
    path,
    reason,
    detail,
}
```

- Support reason categories:
  - no matching rule
  - already organized
  - unsupported metadata
  - unsafe destination
  - destination conflict
  - excluded by policy
- Persist untouched records in the SQLite session store.
- Add count and page-loading APIs for untouched records.
- Extend GUI `PreviewCounts` to include ready, needs review, untouched, and all.
- Add tests for reason classification and persistence.

Acceptance criteria:

- Preview can show an untouched reason breakdown without guessing.
- Existing sessions without untouched records still load safely.
- Ready and needs-review counts retain existing semantics.
- Apply still only moves safe ready operations.

### UX-225-4 - Preview screen redesign

Purpose:

Make Preview the trust center of the app.

Scope:

- Replace large metric cards with a compact summary:

```text
7 files ready to move
18 will stay untouched
No conflicts found
```

- Add untouched reason breakdown:

```text
18 untouched
12 already organized
4 unsupported metadata
2 no matching rule
```

- Show one representative example move inline.
- Add a compact inline planned-changes sample table:
  - File
  - Destination
  - Rule
  - Status
- Keep the full detailed list in a paged window for large folders.
- Add filters:
  - All
  - Ready
  - Untouched
  - Needs review
- Rename the primary Preview action to `Organize N files`.
- Keep secondary actions as `Back`, `Re-analyze`, and `Review all changes`.
- Make AI assistance actionable:
  - show only when AI can analyze, show notes, create draft rules, or is actively running
  - avoid passive or vague "AI is thinking" surfaces unless a task is actually active

Acceptance criteria:

- User can understand what will move, what will stay, and whether conflicts exist in under 10 seconds.
- User can inspect exact source and destination paths through details.
- User can understand why files are untouched.
- The full preview remains paged and responsive on large folders.
- The primary button says exactly what will happen.

### UX-225-5 - Completion and restore redesign

Purpose:

Make successful organization feel safe, final, and reversible without surfacing recovery internals too early.

Scope:

- Redesign completion state around:

```text
Organization complete
99 files moved successfully.
0 skipped · 0 failed
Restore point saved
```

- Actions:
  - `Done` as primary
  - `Restore previous layout` as secondary
  - `View details` as tertiary
- Rename user-facing undo copy to `Restore previous layout`.
- Collapse activity id and journal path under `Technical details`.
- Update confirmation copy to say restore history is recorded before files move.
- Implement `Done` behavior:
  - clear completion action panel
  - keep the same folder selected
  - keep the user in the Organize flow

Acceptance criteria:

- User sees whether the operation succeeded.
- Restore is available immediately but does not visually compete with `Done`.
- Activity id and restore path are hidden by default.
- Failed and skipped files are reported clearly.

### UX-225-6 - Activity timeline

Purpose:

Make Activity a readable audit trail and restore surface instead of a transaction browser.

Scope:

- Replace scope copy with:

```text
Activity for Documents
```

- Add a `Change folder` action that routes back to Organize folder selection.
- Show latest run first with restore availability.
- Group older rows by date.
- Use compact rows:
  - time
  - action summary
  - status dot
  - Details
  - Restore, when available
- Keep restore history and technical details as advanced disclosures.
- Rename row actions from `Undo` to `Restore`.

Acceptance criteria:

- Latest restore opportunity is obvious.
- Activity rows are scannable.
- Technical recovery details remain available but secondary.
- Failed, interrupted, and partially restored activities have clear status language.

### UX-225-7 - Rules screen redesign

Purpose:

Make rule and profile management lighter, more focused, and less administrative.

Scope:

- Add an Active Profile panel at the top:
  - profile name
  - rule count
  - destination example
  - `Edit profile`
  - `Use another`
- Convert built-in profiles to compact rows.
- Convert custom profiles to compact rows.
- Reduce repeated buttons and heavy badges.
- Keep import/export available, but visually secondary.
- Move AI rule generation into a dedicated contextual panel:
  - `Suggest rules from current folder`
  - `Describe how you want files organized`

Acceptance criteria:

- Active profile is identifiable immediately.
- Built-in and custom profiles are easy to compare.
- AI rule generation feels contextual, not bolted onto normal utility actions.
- The Rules screen no longer reads as a list of large bordered cards.

### UX-225-8 - Rule builder refinement

Purpose:

Make powerful custom rules understandable without making the editor feel like raw configuration.

Scope:

- Keep Profile Workspace as an `egui::Window` for the initial v2.25 pass.
- Simplify toolbar:
  - profile name
  - rule count
  - Save
  - Use profile
  - Validate
  - AI or overflow actions
- Move profile ID and technical profile details into Advanced.
- Redesign left rule list:
  - compact rows
  - destination preview
  - slim selected accent
- Redesign selected rule editor into sections:
  - Rule
  - Applies to
  - Destination
  - Conditions
  - Advanced
- Refine the destination builder to read as a path:

```text
images / {year} / {month}
```

- Hide advanced tokens by default.
- Move `Match all files` into Advanced with a warning.
- Add rule simulation after preview reason data exists:

```text
This rule matches 7 files in the current folder.
```

Acceptance criteria:

- The editor is visually calmer.
- Destination paths are easier to understand.
- Match-all behavior is guarded.
- A user can validate a rule before using it.

### UX-225-9 - Settings refinement

Purpose:

Keep settings clear, sparse, and safe.

Scope:

- Regroup Settings into:
  - Appearance
  - AI assistance
  - Privacy
  - Explorer integration
  - Advanced and maintenance
- Move AI endpoint and timeout into Advanced.
- Keep provider, model, and readiness status visible in the main AI section.
- Present sampled text inspection as a privacy callout.
- Keep storage cleanup separate from restore history.

Acceptance criteria:

- Settings feels like product configuration, not an engineering panel.
- AI details are available but not visually dominant.
- Privacy-sensitive options are clearly labeled.
- Cleanup copy states that restore history is not removed.

### UX-225-10 - Screen and component extraction

Purpose:

Make the GUI maintainable after the visual redesign lands.

Scope:

- Extract reusable visual helpers first.
- Then split `main.rs` gradually into screen modules:

```text
crates/smartfolder-gui/src/ui/
  components/
  screens/
    organize.rs
    activity.rs
    rules.rs
    settings.rs
  theme/
```

- Keep `SmartfolderApp` as the state owner during early extraction.
- Use action enums from screen rendering functions back to the app to reduce borrow-checker churn.
- Avoid moving core behavior and UI extraction in the same commit when possible.

Acceptance criteria:

- `main.rs` no longer owns most presentation details.
- Screen modules remain behaviorally equivalent after extraction.
- Tests and build remain green after each extraction slice.
- User-visible behavior does not regress during module movement.

### UX-225-11 - Hidden depth enhancements

Purpose:

Add advanced utility without adding visual clutter to the default workflow.

Scope:

- Add tree preview mode.
- Add command palette.
- Add richer file-level explanations.
- Add profile and rule explanations.
- Add complete rule simulation.
- Add contextual AI suggestions:
  - suggest rules from current folder
  - explain untouched files
  - explain why a file matched a rule

Acceptance criteria:

- Power users gain speed and inspection depth.
- New users are not exposed to these controls by default.
- AI suggestions remain subordinate to deterministic preview, rules, undo, and activity history.

## Testing strategy

Required automated checks:

- `cargo fmt --check`
- `cargo test -p smartfolder-core`
- `cargo test -p smartfolder-cli`
- `cargo test -p smartfolder-gui`
- `cargo build -p smartfolder-gui --release`
- focused tests for untouched reason classification and SQLite persistence
- focused tests for profile editor validation after rule-builder changes

Recommended manual checks:

- fresh app launch with no folder selected
- Explorer/preloaded folder launch
- folder drag-and-drop
- current-folder-only analysis
- include-subfolders analysis
- built-in mode analysis for each mode
- custom profile analysis
- preview summary and detailed preview
- untouched reason breakdown
- organize confirmation
- organization completion
- immediate Restore previous layout
- Activity restore for an older action
- AI disabled state
- AI enabled but unavailable state
- AI available with metadata-only mode
- sampled content inspection privacy callout
- light theme visual review
- dark theme visual review
- target minimum window size review

Disposable-folder proof:

```text
Launch smartfolder
  -> select or preload test folder
  -> choose instructions
  -> Preview
  -> verify ready, untouched, and review counts
  -> Organize N files
  -> Done or Restore previous layout
  -> verify original layout is restored when restore is chosen
```

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| UI cleanup hides safety information | Move safety information to disclosures, details windows, and Activity rather than removing it. |
| Preview promises reasons that core does not model | Implement untouched reason records before final Preview redesign. |
| Large preview becomes slow | Preserve SQLite paging and keep full detailed preview in a paged window. |
| Refactor destabilizes app state | Extract visual helpers first, then screens, using action enums and small commits. |
| Dark theme work degrades light theme | Test both themes after token changes and avoid single-palette assumptions. |
| AI becomes too prominent | Show AI only as actionable assistance after deterministic analysis. |
| Rename from undo to restore obscures meaning | Keep confirmation copy explicit: restore previous layout using restore history. |
| Rule builder scope expands too far | Keep Profile Workspace as a window first and defer full-page workspace migration. |

## Explicitly out of scope for v2.25

- autonomous organization without preview
- disabling confirmation for file moves
- weakening restore history
- replacing the Rust desktop GUI with a web frontend
- replacing deterministic rules with AI classification
- duplicate detection
- broad content extraction or OCR
- enterprise policy management
- installer and auto-update work

## Final release gate

The v2.25 UX redesign is complete when:

- Organize, Preview, Completion, Activity, Rules, Rule Builder, and Settings all use the updated design foundation.
- Preview has real untouched reason data.
- User-facing undo language is consistently replaced with restore language.
- Technical details are collapsed by default.
- The app remains responsive with paged previews.
- The disposable-folder proof passes.
- Automated validation passes:

```text
cargo fmt --check
cargo test -p smartfolder-core
cargo test -p smartfolder-cli
cargo test -p smartfolder-gui
cargo build -p smartfolder-gui --release
```
