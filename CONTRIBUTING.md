# Contributing

`smartfolder` prioritizes correctness, transparency, and data safety over convenience.

## Core Constraints

Before changing behavior, preserve these v1 constraints:

- Do not read file contents.
- Do not move files without preview and explicit confirmation.
- Do not overwrite existing files.
- Do not allow destinations outside the selected root.
- Do not follow symlinks or junctions by default.
- Keep undo and transaction journal behavior reliable.

## Code Documentation Standards

**All new code must be comprehensively documented.** This is mandatory for all pull requests and commits.

### Module-Level Documentation

Every file must begin with a module-level doc comment (`//!`) that explains:

- **Purpose**: What the module does in 1-2 sentences
- **Overview**: High-level description of key types and functions
- **Key features**: Main capabilities and characteristics
- **Example workflow** (if applicable): Shows typical usage patterns

Example:
```rust
//! File system scanning with metadata collection.
//!
//! Recursively traverses a directory tree, collecting metadata about files
//! (names, sizes, timestamps, types) without reading file contents.
//!
//! # Features
//!
//! - Filtering by depth, hidden files, system files, and project folders
//! - Cancellable scans via `CancellationToken`
//! - Detailed warnings for unreadable or skipped entries
```

### Function Documentation

Every public function must have a doc comment that includes:

- **Brief description**: What the function does
- **Logical flow** (for complex functions): Step-by-step explanation of execution
- **Errors**: Which error conditions can occur
- **Example** (for exported APIs): Code showing typical usage

Example:
```rust
/// Generate a plan from a scan result and planning options.
///
/// # Logical Flow
///
/// 1. For each file in the scan:
///    - Apply rule matching to determine destination
///    - Check if destination path is safe
/// 2. Detect conflicts (exists, case-only rename, path too long)
/// 3. Mark conflicting operations as "not selected"
/// 4. Generate summary statistics
/// 5. Return complete plan
///
/// # Errors
///
/// Returns error for IO failures or invalid paths.
pub fn generate_plan(
    root: impl AsRef<Path>,
    scan: &ScanResult,
    options: &PlanOptions,
) -> Result<PlanRecord> {
```

### Struct and Enum Documentation

Every public struct and enum must have a doc comment describing:

- **Purpose**: What the type represents
- **Fields** (for structs): Brief explanation of each field
- **Variants** (for enums): Description of each variant
- **Invariants**: Any guarantees the type maintains

Example:
```rust
/// Complete plan for organizing files in a directory.
///
/// Generated by the planner phase, contains:
/// - List of operations to be executed (each with conflict detection)
/// - Files that couldn't be classified
/// - Warnings and summary statistics
///
/// Plans are serializable to JSON and can be reviewed, edited, and applied later.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct PlanRecord {
    pub schema_version: u16,
    pub plan_id: String,
    pub root: PathBuf,
    /// ... other fields
}
```


### Documentation Style Guidelines

- **Crisp and concise**: No verbose prose; convey information efficiently
- **Clear purpose**: Explain *why* and *what*, not just *how*
- **Complete coverage**: Every public item must be documented
- **Logical flow**: For complex operations, describe the step-by-step execution
- **Error clarity**: Specify what errors can occur and when
- **Examples**: Provide runnable examples for public APIs

### Checking Documentation

Before submitting code:

1. Verify all public items have doc comments
2. Run `cargo doc --no-deps --open` to check rendered documentation
3. Ensure doc comments are grammatically correct and technically accurate
4. Check that logical flows make sense and match the actual implementation

## Build and Test

Run the workspace checks before submitting changes:

```powershell
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo doc --no-deps --lib
```
